## Infraestructura Cloudflare

La empresa ya usa Cloudflare. Todo el proyecto vive dentro de su ecosistema:

- **Cloudflare Pages**: sirve el sitio Astro, CDN global, despliegue automГЎtico desde Git
- **Cloudflare D1**: base de datos SQLite serverless para las submissions de formularios
- **Cloudflare Access**: protege la ruta `/admin` con login por email/Google, sin cГіdigo de autenticaciГіn propio
- Plan gratuito es suficiente para el volumen actual (~107 visitas/dГ­a, picos en matrГ­cula ~1,000/dГ­a)

## Arquitectura de datos

### Contenido (solo lectura, desde APIs externas)
- **Cursos y facultades**: API externa REST (ya existe, no es responsabilidad de este proyecto)
- **Noticias**: estГЎticas en Fase 1 (archivos en `src/data/`), Fase 2 desde API externa
- El frontend hace `fetch()` a estas APIs en build time (SSG) o en runtime segГєn corresponda

### Formularios (escritura, responsabilidad de este proyecto)
- El frontend llama a **Astro Actions** (`src/actions/index.ts`)
- Las Actions validan con Zod y escriben en **Cloudflare D1**
- Una sola tabla `submissions` con columna `form_type` para distinguir formularios
- El admin accede a `/admin` (protegido por Cloudflare Access) para ver y exportar CSV

> **EspecificaciГіn de campos de cada formulario**: ver `CONTENT.MD` в†’ "Formularios del proyecto".

## Schema de base de datos D1

```sql
-- db/schema.sql
CREATE TABLE IF NOT EXISTS submissions (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  form_type   TEXT NOT NULL,
  data        TEXT NOT NULL,  -- JSON con todos los campos del formulario
  created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_form_type ON submissions(form_type);
CREATE INDEX IF NOT EXISTS idx_created_at ON submissions(created_at);
```

DecisiГіn de diseГ±o: los campos del formulario se guardan como JSON en la columna `data`. Esto permite que cada formulario tenga sus propios campos sin necesidad de mГєltiples tablas o columnas opcionales. Para exportar CSV, se parsea el JSON y se generan las columnas dinГЎmicamente.

### `astro.config.mjs`
```js
import { defineConfig } from 'astro/config';
import cloudflare from '@astrojs/cloudflare';
import tailwindv4 from '@tailwindcss/vite';

export default defineConfig({
  output: 'server',          // necesario para Actions y D1
  adapter: cloudflare(),
  vite: {
    plugins: [tailwindv4()],

});
```

> **Nota**: `output: 'server'` es necesario porque las Astro Actions requieren server-side rendering. Las pГЎginas puramente estГЎticas pueden usar `export const prerender = true` individualmente para recuperar el comportamiento SSG y maximizar el SEO.

## Convenciones y reglas del proyecto

1. **TypeScript estricto** en todo el proyecto (`strict: true` en `tsconfig.json`)
2. **Astro Actions** para toda la lГіgica de formularios вЂ” NO crear archivos en `pages/api/`
3. **`export const prerender = true`** en pГЎginas que no necesiten server (inicio, nosotros, cursos estГЎticos) para mГЎximo rendimiento SEO
4. **Zod** para validaciГіn de todos los inputs вЂ” las Actions de Astro lo incluyen nativamente con `defineAction` y `z` de `astro:schema`
5. **Un componente por formulario** вЂ” no un componente genГ©rico con props condicionales
6. **Tailwind v4** вЂ” la configuraciГіn es CSS-first, tokens en `global.css` con `@theme`. **Solo debe haber un `@import "tailwindcss";`** en el archivo global de CSS.
7. **ImГЎgenes estГЎticas del sitio** en `public/images/`, imГЎgenes de contenido dinГЎmico vienen de las APIs externas
8. Las pГЎginas de `/admin` deben tener `export const prerender = false` explГ­cito
9. **Contenido textual** вЂ” replicar fielmente los textos, tГ­tulos y descripciones definidos en `CONTENT.MD`. No inventar ni modificar copy sin autorizaciГіn.

> **Estilo, tipografГ­a y tokens visuales**: toda la jerarquГ­a tipogrГЎfica, tamaГ±os, pesos, tracking, line-height, paleta de colores, clases Tailwind prohibidas/permitidas y reglas de UI estГЎn definidas exclusivamente en `STYLES_GUIDE.MD`. Ni `CLAUDE.MD` ni `CONTENT.MD` definen estilo.
